# AGENTS.md

This file gives coding agents a fast, project-specific map of the `leadtype` repository.

## Project Snapshot

- Module: `github.com/rowland/leadtype`
- Language: Go 1.21
- Primary purpose: PDF generation, font handling, rich text, and LTML document/layout rendering
- Shape of the repo: library-first, with sample programs and a small CLI utility

## Repository Map

- `pdf/`: core PDF writer and document model
- `font/`: shared font abstraction used across AFM and TTF paths
- `ttf/`: TrueType parsing, metrics, and subsetting support
- `ttf_fonts/`: loads TTF font collections from the filesystem
- `afm/` and `afm_fonts/`: Adobe Font Metrics support and built-in AFM data
- `codepage/`: legacy single-byte encoding support for simple font output
- `rich_text/`: rich-text segmentation and rendering helpers
- `ltml/`: XML-based document/layout layer built on top of the PDF stack
- `colors/`, `options/`, `wordbreaking/`: supporting utilities
- `haru/`: alternate PDF-related implementation pieces and tests
- `samples/`: runnable examples that generate PDFs
- `ttdump/`: small CLI for TTF inspection
- `docs/`: design and workflow notes, especially for Unicode PDF work

## How The Pieces Fit Together

- The historical PDF text path is single-byte and codepage-based.
- The newer Unicode work adds composite font output for TTF fonts, `ToUnicode` maps, glyph recording, and subsetting.
- `pdf.DocWriter` is the main orchestration point for document construction and font registration.
- `ltml.Doc` parses LTML XML and prints through the writer stack into PDFs.

When changing text rendering or fonts, expect interactions across `pdf/`, `font/`, `ttf/`, `ttf_fonts/`, and sometimes `rich_text/` or `codepage/`.

## Baseline Workflow

From the module root:

```bash
go build ./...
go test ./...
```

These are the default safety checks for almost any change. Keep the tree buildable.

## Testing Conventions

- Prefer package-level unit tests close to the code you changed.
- Use in-repo fixtures under `testdata/` where possible.
- Avoid adding tests that depend on system fonts unless they skip cleanly when fonts are unavailable.
- PDF golden tests live behind normal comparisons with timestamp normalization in `pdf/super_test.go`.
- Golden PDFs can be regenerated with:

```bash
go test ./pdf -update
```

- The minimal synthetic TTF fixture is generated by:

```bash
go run ttf/testdata/generate/main.go
```

Only regenerate committed fixtures when the fixture spec itself changes.

## Project-Specific Guardrails

- Do not remove or break the existing AFM/simple-font path while working on Unicode/TTF behavior.
- Preserve backward compatibility unless the task explicitly calls for a breaking API change.
- Favor small, reviewable changes. The docs in `docs/development-process.md` expect logically scoped commits and green builds/tests.
- Many sample programs assume macOS conveniences such as `/Library/Fonts` and `open`; treat them as examples, not portable test infrastructure.

## Editing Advice

- Start by locating the narrowest package that owns the behavior.
- For rendering bugs, inspect tests first; this repo already has strong coverage in `pdf/`, `ttf/`, and `ltml/`.
- When changing serialized PDF output, check whether golden tests or explicit string-serialization tests need updates.
- When changing font metrics or glyph mapping, verify both isolated unit tests and any PDF-level integration tests affected by the change.

## Definition Of Done

For most changes, aim to leave behind:

- focused code changes in the owning package
- updated or added tests
- `go build ./...` passing
- `go test ./...` passing, or a clear note about any intentionally skipped or environment-specific cases
